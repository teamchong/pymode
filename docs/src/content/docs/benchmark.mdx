## When you actually pay the Cold number

The Cold column is the `Worker Startup Time` Cloudflare reports right
after `wrangler deploy`. You pay it:

- **Once per fresh deploy** — the first request after a new version
  goes live. Both runtimes pay this.
- **On the next request after isolate eviction** — Cloudflare evicts
  idle isolates after some minutes of no traffic (the exact threshold
  isn't published; it's adaptive). The very next request to that PoP
  triggers a fresh isolate boot.

You do **not** pay it on every request. Once an isolate is warm in a
given PoP, subsequent requests in that PoP go straight to the warm
path. Cloudflare also uses memory snapshots in production for both
runtimes, which makes real cold-isolate spin-ups faster than the
deploy-time number; the deploy-time number is the worst-case
bound.

Warm latency (the p50 / p99 columns) is what dominates steady-state
traffic. For stdlib-only work, pymode and Pyodide land within noise
of each other; for the package-heavy workloads, behaviour depends on
how much work happens on each side of the Python ↔ JS boundary.

**Both runtimes can run popular Python packages.** Earlier versions of
this page claimed `markdown_it_py`, `jinja2`, and `beautifulsoup4`
weren't loadable on Cloudflare's Python Workers. That was true under
the original wrangler flow but **not** under the current
`pywrangler` + `uv` + `WorkerEntrypoint` workflow, which installs
arbitrary pure-Python packages from PyPI. Both bench workers above use
the modern flow.

## Where pymode is meaningfully different

The package-availability gap is closed for pure-Python libraries.
What remains:

- **Vanilla upstream CPython 3.13.** Pyodide ports CPython on a
  ~yearly cadence and maintains its own fork. If you want day-one
  3.14 features (template strings, deferred evaluation, free-threaded
  mode) or want to avoid Pyodide-specific patches around signal
  handling / threading / random / fs semantics, pymode is plain
  CPython from `git`.
- **AOT-specialised host imports.** pymode's deploy CLI generates a
  WASM snapshot tailored to the app's imports and patches hot I/O
  paths (KV, R2, fetch, etc.) into native JS stubs so they bypass
  the Python ↔ JS FFI translation that Pyodide pays per call. The
  cost is a longer deploy; the benefit is lower per-request overhead
  for I/O-bound workers.
- **C extensions Pyodide doesn't ship.** Pyodide carries a curated
  set of compiled extensions (numpy, pandas, pillow, lxml…); pymode
  can compile arbitrary C extensions via `zig cc` against
  `wasm32-wasi`. The long tail of niche or in-house native modules
  is the place this matters.
- **Smaller per-isolate footprint.** Pyodide's base runtime is ~6.4 MB
  before packages; pymode's stripped CPython sits in a wasm binary
  tailored per deploy. Fewer megabytes per isolate means more isolates
  per machine and faster snapshot loads.

If your worker is short, pure-Python, and your packages are already in
Pyodide's catalog, the official Python Workers will serve you well.
pymode is the better fit when you need a specific CPython version, a C
extension Pyodide doesn't carry, or you're willing to trade longer
deploy time for lower per-request overhead.
